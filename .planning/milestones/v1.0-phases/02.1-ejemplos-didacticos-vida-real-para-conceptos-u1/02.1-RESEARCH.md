## Summary

Esta fase es exclusivamente de **autoría de contenido dentro de archivos TSX existentes**. No hay nuevas dependencias, nuevos componentes ni cambios arquitecturales. El trabajo consiste en insertar bloques JSX (Callout, pre, tablas HTML) directamente dentro de las funciones Topic11–Topic14 en `src/content/units/u1/concepts.tsx`.

El stack está completamente establecido desde la Fase 2: React 19, Tailwind v4, lucide-react, y los patrones de tabla/callout ya usados en el archivo. La decisión pendiente más relevante es el syntax highlighting para snippets SQL — el proyecto ya tiene `@uiw/react-codemirror` instalado con `@codemirror/lang-sql`, lo que permite un componente `<SqlSnippet>` readonly de peso mínimo sin instalar nada nuevo.

El mayor riesgo no es técnico sino pedagógico: que los ejemplos sean demasiado largos (rompen el flujo de lectura), demasiado abstractos (pierden el efecto "aha!"), o que interrumpan la estructura visual establecida del contenido teórico. La clave es insertar justo después del concepto que ilustran, con tono diferenciado (coloquial) respecto al contenido formal.

**Primary recommendation:** Editar solo `src/content/units/u1/concepts.tsx`. Para SQL highlighting usar `<pre className="bg-zinc-900 border border-zinc-800 rounded-md p-4 text-xs text-green-300 overflow-x-auto font-mono">` ya establecido en Topic13 linea 702 — no crear un componente CodeMirror readonly, eso es over-engineering para ejemplos visuales estáticos.

## Architecture Patterns

### Estructura del archivo objetivo
```
src/content/units/u1/concepts.tsx
├── Topic11()   # 1.1 Definicion de BD — agregar ejemplos Dato/Info/BD/DBMS
├── Topic12()   # 1.2 Archivos vs BD — agregar ejemplos redundancia/inconsistencia
├── Topic13()   # 1.3 Tipos de BD — agregar ejemplos por criterio seleccionado
└── Topic14()   # 1.4 Proceso de Diseño — agregar ejemplos SI/OLTP/OLAP
```

### Pattern 1: Ejemplo inline post-concepto
**Que es:** Un `<Callout type="example">` insertado justo despues del bloque que define el concepto.
**Cuando usar:** Para cada concepto clave que se beneficia de un escenario concreto.
**Ejemplo:**
```tsx
<Callout type="definition" title="Dato">
  Hechos relacionados con personas, objetos, eventos...
</Callout>

{/* NUEVO: insertar aqui */}
<Callout type="example" title="Dato en la UTFSM">
  Imagina que eres alumno de la UTFSM. Tu RUT, tu rol (201234567-8),
  la nota 5.3 del certamen 1 de INF-239, y la fecha 2026-03-15 son
  <strong>datos</strong> — hechos aislados almacenados en la BD de la universidad.
</Callout>
```

### Pattern 2: Mini-tabla de datos ilustrativos
**Que es:** Una tabla HTML de 3-5 filas mostrando datos concretos de ejemplo.
**Cuando usar:** Para explicar PK/FK, redundancia, estructura relacional.
**Ejemplo:**
```tsx
<div className="overflow-x-auto rounded-md border border-zinc-700 mt-2">
  <table className="w-full text-sm border-collapse">
    <thead>
      <tr className="bg-zinc-800">
        <th className="text-left px-4 py-2 text-zinc-300 font-semibold border-b border-zinc-700">RUT</th>
        <th className="text-left px-4 py-2 text-zinc-300 font-semibold border-b border-zinc-700">NOMBRE</th>
        <th className="text-left px-4 py-2 text-zinc-300 font-semibold border-b border-zinc-700">CARRERA</th>
      </tr>
    </thead>
    <tbody>
      {[
        ['12.345.678-9', 'Ana Soto', 'Ingenieria Informatica'],
        ['23.456.789-0', 'Luis Vera', 'Ingenieria Civil'],
        ['34.567.890-1', 'Maria Paz', 'Arquitectura'],
      ].map(([rut, nombre, carrera], i) => (
        <tr key={rut} className={i % 2 === 0 ? 'bg-zinc-900' : 'bg-zinc-900/50'}>
          <td className="px-4 py-2.5 text-zinc-200 font-mono text-xs border-b border-zinc-800">{rut}</td>
          <td className="px-4 py-2.5 text-zinc-400 border-b border-zinc-800">{nombre}</td>
          <td className="px-4 py-2.5 text-zinc-400 border-b border-zinc-800">{carrera}</td>
        </tr>
      ))}
    </tbody>
  </table>
</div>
```

### Pattern 3: Snippet SQL estatico
**Que es:** Un `<pre>` con font-mono verde, identico al ya usado en Topic13 linea 702.
**Cuando usar:** Para mostrar SQL de referencia visual (SELECT, INSERT, CREATE TABLE).
**Ejemplo:**
```tsx
<pre className="bg-zinc-900 border border-zinc-800 rounded-md p-4 text-xs text-green-300 overflow-x-auto font-mono">
{`SELECT RUT, NOMBRE, PROMEDIO
FROM ALUMNO
WHERE PROMEDIO > 5.0
ORDER BY PROMEDIO DESC;`}
</pre>
```

### Anti-Patterns a Evitar
- **Ejemplo antes del concepto:** Los ejemplos van DESPUES de la definicion, no antes.
- **Tono formal en ejemplos:** Los Callout definition usan tono academico; los Callout example deben ser coloquiales — si suena a apunte de clase, reescribir.
- **Ejemplos truncados:** Decision locked — siempre completos, nunca "... ver mas".
- **Nuevo tipo de Callout:** Decision locked — usar solo `type="example"` existente.
- **Collapsible/accordion:** Decision locked — ejemplos siempre visibles.
- **CodeMirror para SQL snippets:** Over-engineering. El `<pre>` ya en uso es suficiente.
- **Modificar ConceptsSection.tsx o Callout.tsx:** Solo se edita `concepts.tsx`.

## Common Pitfalls

### Pitfall 1: Ejemplo que repite la definicion en otras palabras
**Que pasa:** El estudiante lee el ejemplo y no aprende nada nuevo — solo ve la misma idea con distintas palabras.
**Por que ocurre:** Es facil "ilustrar" un concepto parafraseandolo.
**Como evitar:** Cada ejemplo debe aportar algo concreto: un nombre real (empresa/persona), un numero real, una situacion especifica que no este en la definicion teorica.
**Warning signs:** Si el ejemplo podria escribirse sin mencionar ninguna empresa, persona o dato concreto, es demasiado abstracto.

### Pitfall 2: Ejemplo demasiado largo que rompe el flujo de lectura
**Que pasa:** El estudiante pierde el hilo — no puede distinguir donde termina el ejemplo y donde continua la teoria.
**Por que ocurre:** Querer ser exhaustivo en el ejemplo.
**Como evitar:** Ejemplos cortos (2-6 lineas de texto). Si necesita mas, usar sub-bullets o mini-tabla en lugar de prosa larga.
**Warning signs:** Si el Callout tiene mas de 8 lineas de texto continuo.

### Pitfall 3: Tono inconsistente — ejemplo suena academico
**Que pasa:** El contraste formal/coloquial se pierde — el estudiante no percibe el cambio de registro.
**Por que ocurre:** Inercia del tono del contenido teorico circundante.
**Como evitar:** Usar segunda persona ("Imagina que..."), nombrar empresas directamente ("Netflix usa..."), escribir como si estuvieras explicando verbalmente en clases.

### Pitfall 4: Acumulacion de ejemplos sin seleccion
**Que pasa:** Cada concepto tiene 3-4 ejemplos — la pagina se vuelve infinita.
**Por que ocurre:** Dificultad para elegir el mejor ejemplo.
**Como evitar:** Regla de oro: un ejemplo por concepto clave. Solo agregar segundo si los dos escenarios (universitario + tech) aportan perspectivas genuinamente distintas que el primero no cubre.

## Open Questions

1. **Syntax highlighting de SQL en ejemplos nuevos**
   - Lo que sabemos: El proyecto tiene `@uiw/react-codemirror` + `@codemirror/lang-sql` instalados. Topic13 usa `<pre>` verde sin highlighting real.
   - La duda: La CONTEXT.md marca esto como "Claude's Discretion".
   - Recomendacion: Usar `<pre>` identico al de Topic13. Un componente CodeMirror readonly añade complejidad sin beneficio pedagogico real para ejemplos de 3-5 lineas. Si el usuario quiere ejecutar SQL, ya tiene la seccion Editor SQL.

2. **Cantidad de ejemplos en Topic13 (6 criterios de tipos de BD)**
   - Lo que sabemos: 6 criterios, algunos muy tecnicos (numero de procesadores, numero de sitios).
   - La duda: Criterios 4-6 son muy tecnicos y tienen poco impacto en un curso introductorio.
   - Recomendacion: Ejemplos solo en Criterios 1 (estructura), 2 (nivel org.), y 3 (tipo de dato). Criterios 4-6 pueden tener ejemplos muy cortos inline en el texto existente o ninguno.
